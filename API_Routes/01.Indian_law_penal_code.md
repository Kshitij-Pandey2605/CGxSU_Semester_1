# ⚖️ Indian Penal & Legal API Documentation

This API serves as a comprehensive digital repository for Indian laws, sections, and legal precedents. It is designed for practicing complex backend architecture, including advanced filtering, nested data aggregation, role-based access control (RBAC), and high-performance data streaming.

---

## 🛠 Basic CRUD Operations

*Standard resource management for legal entries.*

* `GET /api/v1/laws` - Fetch a paginated list of all legal sections.
* `GET /api/v1/laws/:id` - Retrieve complete details of a specific legal section.
* `POST /api/v1/laws` - Create a new legal entry (Requires admin privileges).
* `PUT /api/v1/laws/:id` - Replace an entire legal entry with updated data.
* `PATCH /api/v1/laws/:id` - Partially update a legal entry (e.g., updating just the punishment details).
* `DELETE /api/v1/laws/:id` - Permanently remove a legal section from the database.
* `GET /api/v1/laws/:id/exists` - Lightweight check to see if a specific Law ID exists.

## 🔍 Search & Discovery

*Advanced text processing and pattern matching routes.*

* `GET /api/v1/laws/search?q=query` - Full-text search across titles and descriptions.
* `GET /api/v1/laws/search/autocomplete?q=murd` - Returns top 5 suggestions for search bars.
* `GET /api/v1/laws/search/fuzzy?q=theft` - Fuzzy matching to account for typos in legal terms.
* `GET /api/v1/laws/search/exact?section=302` - Exact match search for specific section numbers.
* `GET /api/v1/laws/search/starts-with?prefix=pun` - Find laws where the title starts with a specific string.
* `GET /api/v1/laws/search/ends-with?suffix=ment` - Find laws where the title ends with a specific string.
* `GET /api/v1/laws/search/case-insensitive?q=IPC` - Search without case sensitivity.
* `GET /api/v1/laws/search/highlight?q=fraud` - Returns search results with matching snippets highlighted.
* `GET /api/v1/laws/search/regex?pattern=^Section\s[0-9]` - Advanced regex-based search for legal formatting.
* `GET /api/v1/laws/search/synonyms?q=robbery` - Search including related legal terms like "dacoity."

## 📂 Filtering & Categorization

*Deep-dive filtering based on legal hierarchies and metadata.*

* `GET /api/v1/laws/filter/chapter/:chapterId` - Fetch all laws belonging to a specific chapter.
* `GET /api/v1/laws/filter/act/:actName` - Filter by specific Act (e.g., IPC, CrPC, BNS).
* `GET /api/v1/laws/filter/status?active=true` - Filter by current legal validity.
* `GET /api/v1/laws/filter/punishment?type=imprisonment` - Filter by nature of punishment.
* `GET /api/v1/laws/filter/fine?min=500&max=5000` - Filter by the range of monetary penalty.
* `GET /api/v1/laws/filter/bailable?value=true` - Filter by bailable vs. non-bailable offenses.
* `GET /api/v1/laws/filter/cognizable?value=false` - Filter by cognizable vs. non-cognizable status.
* `GET /api/v1/laws/filter/tags?list=cyber,fraud` - Filter by a comma-separated list of tags.
* `GET /api/v1/laws/filter/date?after=2023-01-01` - Laws enacted or updated after a certain date.
* `GET /api/v1/laws/filter/updated-recently` - Returns laws modified in the last 30 days.
* `GET /api/v1/laws/filter/bookmarks` - Retrieve laws bookmarked by the current user.
* `GET /api/v1/laws/filter/trending` - Laws with the highest view counts this week.
* `GET /api/v1/laws/filter/complexity?level=high` - Filter by subjective difficulty or legal complexity.
* `GET /api/v1/laws/filter/court-type?name=HighCourt` - Laws often cited in specific courts.
* `GET /api/v1/laws/filter/offense-type?name=Homicide` - Grouping by the nature of the crime.

## ✨ Good to Have (Intermediate Backend Features)

* **Request Validation Middleware:** Use **Express-Validator** or **Zod** to ensure that inputs like `section` are numbers and `title` is not empty before processing.
* **Response Standardization:** Every route should return a consistent structure:
```json
{ "status": "success", "data": { ... }, "meta": { "total": 100 } }

```


* **JWT Expiry & Rotation:** Implement short-lived access tokens and longer-lived refresh tokens for security.
* **Bcrypt Hashing:** Ensure all user passwords are encrypted using **Bcrypt** with at least 10 salt rounds.
* **Rate Limiting:** Protect your `/search` and `/auth` routes from brute force attacks using **express-rate-limit**.
* **CORS Policy:** Configure Cross-Origin Resource Sharing to only allow your specific frontend domain to access the API.
* **Compression:** Use the **compression** middleware to shrink the size of JSON responses for better mobile performance.
* **Environment Variables:** Store all sensitive data (DB_URL, JWT_SECRET) in a `.env` file and never commit it to Git.
* **Centralized Error Handler:** Use a single middleware to handle all `404` and `500` errors, ensuring no technical stack traces are leaked to the user.
* **Database Indexing:** Ensure fields like `lawName`, `Section`, and `tags` are indexed in your database to keep search speeds fast.
